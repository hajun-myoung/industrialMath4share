### DOMContentLoaded

이 이벤트는 "Document(문서)"의 내용이 준비되었을 때 발생합니다. ('준비'에 대한 자세한 기준은 아래 [이벤트 판단 기준](./#이벤트-판단-기준)을 참고해주세요)
여타 이벤트들이 "클릭할 때", "키보드를 눌렀을 때"와 같이 조건을 가지고 있다는 점을 생각하여 보면, 이 이벤트의 조건이 **문서의 모든 내용이 준비되었을 때**에 해당하는 것입니다.

그런데, 이러한 조건은 웹 생태계 상에서 아주 중요한 역할을 차지합니다.

### 오류 상황: Early Accessing

많은 경우, 다음과 같은 querySelector를 사용하여 스크립트를 구현합니다:

```js
const e = document.querySelector('#id');
// 또는
const e = document.getElementById('id');
// 또는
const e = document.getElementsByClassName('class');
```

그런데, 이러한 접근은 "아이디가 ~인", "클래스가 ~인"을 조건으로 사용하는 _CSS Selector_ 를 사용하고 있습니다. 즉, 그러한 요소가 존재하지 않는다면 의도한 변수 `const e` 에는 `null`이 할당될 것이고, 이는 곧 하위의 모든 스크립트가 오류가 나게끔 합니다.

예를 들어, `e`가 특정한 `div` 요소였기를 바랐고, 이 요소의 내용에 사용자 이름을 넣어 표시하고자 했다면:

```js
const username = await getUserName();
// username을 로딩하는 가상의 함수(async)
const e = document.getElementById('username');
// e = null 할당
e.innerText = username;
// e가 null, innerText 프로퍼티를 가지지 않음
// Uncaught TypeError: Cannot read properties of null (reading 'innerText')
```

위와 같이 "null의 innerText라는 프로퍼티를 읽을 수 없다"는 오류가 발생하게 될 것입니다

### 해결 방법: 이벤트의 사용

이러한 경우, `DOMContentLoaded` 이벤트리스너의 사용이 좋은 방법으로 떠오릅니다. 이 이벤트는 상술한 것과 같이 "문서가 모두 준비되었을 때" 발생합니다. 따라서, 문서 내의 요소들이 "누락"될 걱정 없이, **_로딩 후에 하위 스크립트를 수행_** 시키는 것이 가능합니다.

### 이벤트 판단 기준

그런데, "문서의 모든 내용이 준비되었다"는 표현은 어딘가 애매합니다. 이는 "준비"의 기준이 공유되지 않고 있기 때문인데, WebAPI는 이를 다음과 같이 해석합니다:

> The DOMContentLoaded event fires when the HTML document has been completely parsed, and all deferred scripts (`<script defer src="…"> and <script type="module">`) have downloaded and executed. It doesn't wait for other things like images, subframes, and async scripts to finish loading.  
> [MDN Document: Document: DOMContentLoaded event](https://developer.mozilla.org/en-US/docs/Web/API/Document/DOMContentLoaded_event)

이를 정리하면, 다음과 같습니다:

1. HTML 문서의 파싱이 완료될 것
1. 모든 defer 스크립트 파일들과 module type인 스크립트 파일들이 "다운로드"되고, "실행"되었을 것

#### HTML 문서 파싱의 완료

파싱 완료는, 데이터를 정리함이 끝났다고 해석하면 됩니다. 즉, Object와도 같은 DOM(Document Object Model)의 요소들(각 태그들)이 로딩되었고, 이들의 구조가 정리되어, 접근할 수 있는 상태가 되었음을 의미합니다

#### 스크립트의 다운로드와 실행

기본적으로 이 부분은 세 가지로 나누어 접근하면 이해가 쉽습니다.

1. `defer script`
   - 기본적으로 HTML 문서는 "HTML 문서 로딩 → 스크립트 태그 발견 → HTML 일시 멈춤 → 스크립트 다운로드 및 실행 → 나머지 HTML 로딩"의 순서를 지킵니다
   - 그런데, `defer script`는 조금 다르게 움직입니다: HTML 로딩 → defer script 발견 → 지속적으로 HTML 로딩하면서, defer script 다운로드 → HTML 로딩이 끝나고 실행
   - 즉, 이 부분은 "그렇게 HTML 로딩이 끝나기로 실행하기로 약속한 스크립트까지의 실행이 끝날 것"을 요구하고 있습니다
1. `type=module script`
   - 이 경우는, `module type` 이 지정된 스크립트 파일들을 특정합니다
   - 이것이 지정되면, 다른 스크립트 파일을 `import foo from "bar"` 의 형태를 통해 임포팅하는 것이 가능합니다
   - 즉, 이와 같은 의존성 파일들도 모두 로딩하기를 요구합니다

#### 주의할 점

DOM Parsing, Script Download and Executing이 요구되었지,
모든 내용의 로딩이 요구되지는 않았습니다.  
즉, MDN 문서에서도 말하는 것처럼, 문서 내에 첨부된 이미지 파일들, 서브프레임들, 그리고 async 스크립트들의 응답까지를 기다리지는 않습니다.

## FAQ

### 이 레포지토리는 왜 JS 파일들이 페이지들보다 많나요

JS와 같은 스크립트 파일들은 목적에 맞게 나누었을 때 더 읽기 쉽기 때문입니다  
다음과 같은 구조를 따르고 있고, 목적은 주석에 설명되어 있습니다

```sh
homeworks/0409/js
├── pages # 페이지에 대한 스크립트를
│   ├── signin.js # 로그인 스크립트
│   ├── signup.js # 회원가입 스크립트
│   └── welcome.js # 환영페이지 스크립트
└── util
    ├── auth.js # 사용자 인증(로그인 관련) 스크립트
    ├── storage.js # 사용자의 저장과 저장된 사용자 읽기 등 사용자 관련 스크립트
    └── validation.js # 비밀번호 8자 이상 등 무결성 검증용 스크립트
```

### 저도 JS 파일들을 "서로 참조" 시키고 싶어요

```js
import * as auth from '../util/auth.js';
```

위와 같은 구문을 쓰고 싶다면, HTML 파일에 `<script>` 태그에 `src` 속성 외에도
`type="module"` 이 선언되어 있어야 합니다

#### 그게 뭔데요

> 참조  
> 모듈은 단지 파일 하나에 불과합니다. 스크립트 하나는 모듈 하나입니다.  
> by. [javascript.info/모듈 소개](https://ko.javascript.info/modules-intro)

`type="module"` 속성은 스크립트 파일을 기본 방식 대신 `ECMAScript module` 방식으로 읽게 합니다

"각각의 스크립트 파일이 하나의 모듈" 이므로, 이 모드를 활성화해야 "서로 다른 스크립트들이 서로를 참조"할 수 있어요

#### 참조 어떻게 하는건데요

우선, 참조 대상을 알려주세요

`A.js` 에서 fn1 이라는 함수를 만들었고, 이를 `B.js` 에서 사용하고 싶은 상황이라 가정하겠습니다. 이 경우:

```js
//A.js
export function fn1(...){
^^^^^^ 이렇게 function 앞에 export가 붙어있어야 해요
    ...
}
```

```js
//B.js

// A.js에서 fn1을 직접 참조하여 사용하기
import {fn1} from "./A.js" // 참조
fn1(...) // 사용

// A.js를 통째로 참조해서 사용하기
import * as A from "./A.js" // 전체참조(별명 필수)
A.fn1(...) // 사용
```

### 검증은 대체 어떻게 하는건가요

그것은...생각보다 간단합니다:

1. 조건을 정리하세요
1. 구현하세요

진짜로 이게 다에요

예를 들어서:

```js
/**
 * 경로: /js/util/validation.js
 */
export function ValidatePassword(password) {
  if (!password) {
    throw new Error('비밀번호가 입력되지 않았습니다');
  }

  if (password.length < 8) {
    // 이 조건문을 보세요. 그냥 password 받아서 길이가 8 미만인지 보는게 다에요
    throw new Error('비밀번호는 8자 이상 입력하세요');
  }

  ...
}
```

#### throw new Error가 뭐에요

이건 수업 때 다루어지지 않았어요. 이렇게 구현할 필요 없어요.

보다 간단한 방법으로는, 다음과 같이 구현하시면 돼요:

```js
//폼 버튼을 누르면...
if (password.length < 8) {
  alert('비밀번호는 8자 이상 입력하세요'); // 알림창을 띄워서 거부
  // 로컬스토리지에 저장하는 로직이 "없으면" 됨
}
```

### hashString이 뭐에요

오호 이걸 물어보다니, 제 프로젝트를 자세히 뜯어보고 이해하셨군요!

> 아니면 GPT 돌렸거나...

이건 비밀번호를 _조금 더 까리하게_ 저장하는 방법이에요

우리가 배운 기본 방식은 "로컬스토리지에 아이디와 비번을 저장"하는데,
실제로는 이렇게 돌아가는 시스템이 잘 없어요.  
이는, "만약에 회원정보를 저장한 서버가 해킹당하면", 비밀번호까지 유출되기 때문이에요.

그래서 비밀번호를 "암호화한 결과"를 저장하게 되는 것이고, 그 방법이 hashString 이에요.

예를 들어서, 비밀번호가 "1q2w3e4r!" 라면, hashString("1q2w3e4r!")을 한 결과는
"febd93f04bda1aec0d374f8fd014d062525934feb1f1b81ee7c64d61f66b84b1"에 해당해요.

1. 원래 비밀번호는 온데간데 없고,
1. 이걸 안다고 원래 비밀번호를 알아낼 수도 없고
1. 비밀번호에서 한 글자만 달라져도 이 결과는 굉장히 많이 변해서 예측도 안돼요

이렇게 하면, "해킹당해도 비밀번호는 유출되지 않게"할 수 있죠

#### 근데 그러면 로그인도 못하잖아요

맞아요, 이렇게 저장하면 "사용자가 로그인 때 비밀번호를 입력"해도 비교할 수가 없죠  
그래서 다음과 같은 방법을 사용해요:

1. 로그인 때 비밀번호 입력
1. 회원가입 때와 같은 방식으로 비밀번호를 암호화
1. 암호화 된 결과와, 저장된 "암호화 된 비밀번호"를 비교
1. 일치하면 로그인 허락

#### 비밀번호를 까먹었다고 하면요?

아쉽게도, 이건 진짜 복구 불능이에요  
애초에 원래 비밀번호로 돌아가지 못하게(="복호화하지 못하게") 하는 알고리즘이라서 그래요

그렇다보니 요즘 웹앱들은 대개 "비밀번호 초기화/재설정"을 지원하지, "비밀번호 찾기"를 지원하지 않는 거에요
